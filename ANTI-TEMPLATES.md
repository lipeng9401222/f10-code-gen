# ANTI-TEMPLATES · 故意不预制清单

> **铁律**：结构可以预制，**业务内容禁止预制**。
> 此清单是 skill 防漂移的压力器。**清单越长，说明 review 越严肃**。

---

## 为什么需要这份清单

老版本 skill 让 Agent 实时生成所有内容（rules / workflows / 模板），结果：
- 同一个协议生成 5 次出 5 个版本
- Agent 在压力下漏写关键段（Red Flags / Auto-Triggers）
- 业务示例变成"硬编码"，下游项目复用时照抄不思考

新版本：**结构用 cp -R + sed 替换**（机械动作）；**内容必须项目特定**（FILL 标记逼迫思考）。

下面这些**故意没有放进 skill**，理由都写明白了。新增内容前先回答这两个问题：

1. "一个 F10 招投标项目和一个 F10 政务办公项目都会复制这份模板。它们会同意这块内容吗？"
2. 不会 / 大概率不会 → 项目特定，**绝不进 templates/**，必须用 `<FILL: ...>` 占位或挪进本清单

## 已经被故意挡在外面的内容

### A. 业务示例硬编码

| 故意不预制 | 理由 |
| --- | --- |
| ❌ 默认 `mock/list.mock.ts` 写"标段管理"具体字段 | 业务字段是项目级，下游会照抄不改 |
| ❌ 默认列表页 .vue 写死"标段编号/采购项目名称" | 同上 |
| ❌ 默认数据模型 import 路径写 `@epoint-fe/sso-...` 业务包 | 业务包名因项目而异 |
| ❌ 默认 mock 数据用真实政务/采购语料 | 易脱敏失败 |

### B. 团队偏好

| 故意不预制 | 理由 |
| --- | --- |
| ❌ 默认 commit message 格式（"feat: xxx"）当成强制 | 各团队偏好不同，已有 `rules/project-rules.md` 引用项目自己的 CLAUDE.md |
| ❌ 默认 ESLint / Prettier 规则 | 仓库自带，skill 不应该覆盖 |
| ❌ 默认 IDE 配置（.vscode/settings.json） | 个人偏好 |
| ❌ 默认 git hook（pre-commit / commit-msg） | 团队级决策 |

### C. 框架版本耦合

| 故意不预制 | 理由 |
| --- | --- |
| ❌ 默认 `package.json` 锁住 `@epframe/eui-core@9.5.4` 具体版本 | 框架升级时全 skill 失效 |
| ❌ 默认 `eui-cli init` 选择某个产线 | 产线随时间变化 |
| ❌ 默认 nexus 私有源端口（强制 8081） | 各产线 nexus 地址不同 |

### D. 经验性坑点（必须真实换来的）

| 故意不预制 | 理由 |
| --- | --- |
| ❌ 预填的"常见坑点"放进 `references/gotchas.md` | **坑是真金白银调试换来的，不是想出来的**。gotchas.md 启动时**只允许**两个来源：vue-docs 的 troubleshooting.md + frontpage.md ⚠️ 强制段。其他全是 FILL，等真实失败沉淀 |
| ❌ Rationalizations 借口表凭空扩写 | 稀释真实借口的压力值。**只能从真实失败抄** |
| ❌ Red Flags 列表无限加项 | 加多了 Agent 视而不见。只保留最高优先级 6 条 |

### E. 跨平台二次抽象

| 故意不预制 | 理由 |
| --- | --- |
| ❌ 把 `pnpm install` 包装成 `skill:install` 自定义命令 | 抽象一层就有人忘了原命令是啥 |
| ❌ 用 Docker 包装 Node 环境 | 公司大部分人不会 Docker，反而增加门槛 |
| ❌ 写一个统一的"F10 项目脚手架"覆盖 eui-cli | eui-cli 本身就是脚手架，重复造轮子 |

### F. 通用编程知识

| 故意不预制 | 理由 |
| --- | --- |
| ❌ "SQL 注入是坏事"、"XSS 防护"等通用安全教程 | 模型已经知道，写了浪费 token |
| ❌ "Vue 响应式原理"、"Composition API 介绍" | 通用知识 |
| ❌ "什么是 RESTful"、"HTTP 状态码" | 通用知识 |

> 判断标准：**"资深前端工程师第一次看 F10 项目，什么会让他踩坑？"** 那个东西才值得写进 skill。

### G. 过度指令化

| 故意不预制 | 理由 |
| --- | --- |
| ❌ "按钮使用 `bg-blue-500 hover:bg-blue-700` class" | 设计 token 升级后全失效。给约束（"用项目设计 token"）+ 上下文（"见 style-rules.md"），不给具体值 |
| ❌ 默认 form 字段顺序、间距像素值 | 设计规范级，应该走设计系统而不是 skill |

## Drift 检查清单（每季度跑一次）

用两个**真实不同类型**的 F10 项目跑同一个 prompt（如"生成标段管理列表页"）：
- 项目 A：政务办公（appname=govoffice）
- 项目 B：医疗 HRP（appname=medhrp）

`diff -r` 对比生成结果：

| 应该几乎一样的（骨架） | 应该完全不同的（业务内容） |
| --- | --- |
| `ep-layout-manager` 三段式结构 | 列名 / 字段 / mock 数据 |
| `defineDataModel` + `useTableModel` 写法 | 接口路径 / 业务字段 |
| `$dialog` API 调用方式 | 弹窗内表单字段 |
| 工具栏 e-toolbar 五段插槽 | 按钮文字 / 业务操作 |

如果**业务内容也一样** → 说明 templates/ 越界了，必须把项目特定内容降级为 `<FILL>` 或挪进本清单。

## 如果你想往 skill 里加东西

**先回答这两个问题**：

1. **"两个真实 F10 项目都会同意这块内容吗？"**
   - 是 → 协议/骨架，可以进 `rules/` / `workflows/` / `templates/`
   - 否 → 项目特定，必须用 `<FILL: ...>` 占位 + 在本清单加一条"为什么不预制"

2. **"这是真金白银调试换来的吗？"**
   - 是（真实坑点）→ 进 `references/gotchas.md`，**且必须有 1 个具体场景 + 1 个根因 + 1 个解决方案**
   - 否（凭空想象的坑）→ **不要写**，等它真的发生
